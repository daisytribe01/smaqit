## Painpoints Identified

### 1. Test Workflow Interruption

**Category:** Process  
**Severity:** Medium

**Issue:** Test session was redirected to create task documentation (Task 067) before completing the actual validation workflow.

**Impact:** Unable to validate any of the 11 changelog items from v0.6.0-beta release.

**Recommendation:** Complete the E2E workflow execution in a dedicated session using Task 067 as the guide.

### 2. System Actor Pollutes Business Layer with Functional Scope

**Category:** Framework Design  
**Severity:** High

**Issue:** The System actor pattern allows functional and stack requirements to creep into the Business layer. During Business spec generation, the agent included system-level properties (availability, error handling, console output format) that belong in Functional or Stack layers.

**Impact:**
- Business layer loses clarity as stakeholder-focused intent layer
- Functional requirements disguised as business goals
- Stack concerns (console output, terminal support) appearing in business specs
- Layer independence principle violated
- Harder to trace where requirements originate

**Evidence:**
- Business spec contains "System displays colorful console output" (Stack concern)
- Business spec mentions "graceful degradation on unsupported consoles" (Functional behavior)
- Alternative flows include implementation details (ASCII art rendering, monochrome fallback)

**Recommendation:**
1. **Remove System Actor entirely** from Business layer - stakeholders don't think about "the system," they think about user goals
2. **Strengthen Business Agent directives** to explicitly reject functional behaviors, technical properties, and stack concerns
3. **Add validation rule**: Business specs MUST NOT contain words like "display," "output," "render," "console," "terminal," "format" (these indicate functional/stack leakage)
4. **Review LAYERS.md Business directives** to make scope boundaries more explicit and enforceable

### 3. Cross-Layer Duplication and Overlap

**Category:** Framework Design  
**Severity:** Medium

**Issue:** Requirements and acceptance criteria duplicate across layers with varying levels of detail, creating redundancy and maintenance burden.

**Impact:**
- Same requirement appears in multiple layers (e.g., "display ASCII art" in Business and Functional)
- Unclear which layer owns which aspect of functionality
- Updates to requirements require changes in multiple specs
- Testing same capability at multiple layers

**Evidence:**
- Business `BUS-MARIO-GREETING-001`: "Application displays ASCII art representation of Mario character"
- Functional `FUN-MARIO-GREETING-001`: "Application outputs ASCII art representation of Mario character as array of text lines"
- Both testing same capability with different wording

**Recommendation:**
1. Business layer should test stakeholder value ("User sees recognizable Mario greeting"), not implementation artifacts ("ASCII art displays")
2. Functional layer owns behavioral verification
3. Update layer guidance to clarify responsibility boundaries for acceptance criteria

### 4. Incorrect Reference Pattern Usage

**Category:** Framework Compliance  
**Severity:** Medium

**Issue:** Stack spec uses "Implements" reference type when it should use "Enables" for foundation specs.

**Impact:**
- Violates Foundation Reference pattern from LAYERS.md
- Misleading traceability (implies 1:1 mapping when it's 1:many foundation)
- Inconsistent reference semantics across specs

**Evidence:**
- Stack spec: `### Implements - [FUN-MARIO-GREETING]...`
- Should be: `### Enables` (foundation serving multiple potential features)
- Missing Business layer reference entirely

**Recommendation:**
1. Strengthen Stack agent directive to distinguish feature vs foundation specs
2. Foundation specs (base technologies) MUST use Enables reference type
3. Foundation specs SHOULD reference all layers they enable (Business + Functional + Stack)
4. Add validation rule to check reference type matches spec category

### 5. Acceptance Criteria Implementation Leakage

**Category:** Layer Separation  
**Severity:** Low

**Issue:** Stack acceptance criteria test implementation execution instead of stack specification.

**Impact:**
- Stack layer testing implementation details (execution model)
- Blurs line between specification and implementation validation
- Criteria like "Application can execute as Python script" belong in Functional or Coverage testing

**Evidence:**
- Stack `STK-CONSOLE-APP-011`: "Application can execute as a Python script without compilation step"
- This tests runtime execution, not technology selection

**Recommendation:**
1. Stack criteria should verify technology choices are documented, not execution outcomes
2. Execution testing belongs in Functional (behavior) or Coverage (validation) layers
3. Stack criteria format: "Technology X version Y+ is specified" not "Application executes using X"
