## Problems Solved

### 1. Coverage Prompt Violation (Critical)
**Problem:** Test case file instructed Coverage prompt to include performance benchmarks, security requirements, and verification requirements as NEW acceptance criteria, violating the dual-input Coverage model where Coverage reads criteria from upstream specs.

**Detection:** User manually reviewed Coverage section before execution and invoked session.assess.prompt.md workflow, identifying the violation immediately.

**Root Cause:** Test case file (mario-hello.md) and testing agent instructions dated from Dec 2025, predating the dual-input Coverage model implementation (Task 050, Jan 2026).

**Solution:**
- Fixed `docs/test-cases/mario-hello.md` Coverage section to include only test metadata (scope, environment, integration points, thresholds)
- Updated `.github/agents/smaqit.user-testing.agent.md` Phase 3 instructions to clarify Coverage input expectations
- Rationale documented: Coverage specs must read acceptance criteria from upstream specs, not create new ones

**Impact:** Prevented contamination of Coverage spec with criteria not present in upstream layers, preserving layer independence and dual-input model integrity.

### 2. Checkbox Reset Feature Not Tested
**Problem:** Initial E2E workflow only tested new development, not iterative requirement modifications. Task 060 (checkbox reset on modification) remained unvalidated.

**Detection:** User identified after completing Phase 3: "we didnt test iterative development, there is a story to validate, where checkboxes should reset with new iterations"

**Solution:**
- Proposed Luigi character expansion as iterative scenario
- User executed Business agent with expanded requirements (Mario-only → Mario+Luigi random selection)
- Verified checkbox behavior: All 9 criteria reset from [x] to [ ]
- Verified frontmatter: status: draft, added modified timestamp, updated prompt_version
- Updated test report with Luigi iteration validation results

**Impact:** Successfully validated Task 060 feature, bringing total validated changelog items from 6/11 to 7/11 (64%).

### 3. System Actor Pollution Identified
**Problem:** Business specs contained functional and stack concerns disguised as business requirements (e.g., "System displays colorful console output", "graceful degradation on unsupported consoles").

**Root Cause:** System Actor pattern allows technical concerns to creep into stakeholder-focused layer. Stakeholders don't think about "the system" - they think about user goals.

**Solution:** Documented as Painpoint #2 (High severity), created Task 068 for framework fix.

**Recommendation:** Remove System Actor entirely from Business layer, strengthen agent directives to reject behavioral verbs and technical artifacts.

## Key Learnings

### Coverage Dual-Input Model Validation Critical
The manual assessment that caught the Coverage prompt violation demonstrated the importance of reviewing test cases against current framework design. The dual-input model (test metadata from prompt + acceptance criteria from upstream specs) is a critical architectural pattern that must be preserved in all testing workflows.

### Checkbox Reset Requires Iterative Scenario
Testing checkbox reset behavior (Task 060) requires an iterative development scenario where existing validated specs are modified to expand scope. The Luigi expansion successfully demonstrated this feature, showing all checkboxes reset from [x] to [ ], frontmatter reverted to draft, and modification metadata captured.

### System Actor Enables Technical Pollution
The System Actor pattern in Business specs creates a pathway for functional and stack concerns to leak into the stakeholder layer. Removing this actor and strengthening directive language will improve layer separation and framework compliance.

### Reference Pattern Clarity Needed
Both Stack and Infrastructure agents use incorrect reference patterns (Stack missing Business layer references, Infrastructure using Implements instead of Enables). Foundation specs serve multiple layers and should use Enables reference type consistently.

### Manual Verification Limitations
Without CLI execution logging, validating that agents execute commands in the correct order requires manual verification or inference from outcomes. Adding execution logging would improve testing confidence and transparency.
