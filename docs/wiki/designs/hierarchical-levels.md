# Hierarchical Levels

## Overview

smaqit is organized into four hierarchical levels, each building upon the previous. This design decision ensures clear separation of concerns, predictable dependencies, and systematic evolution of the framework.

## The Four Levels

```
Level 0: Framework (Foundation) → Rules and principles
Level 1: Templates (Structure) → Structure definitions
Level 2: Agents & Artifacts (Instances) → Concrete implementations
Level 3: Application (Output) → Working software
```

Each level consumes the level above it and produces inputs for the level below it.

## Level 0: Framework (Foundation)

**What it contains:**
- `framework/SMAQIT.md` — Core principles
- `framework/LAYERS.md` — Layer definitions
- `framework/PHASES.md` — Phase workflows
- `framework/TEMPLATES.md` — Template rules
- `framework/AGENTS.md` — Agent behaviors
- `framework/ARTIFACTS.md` — Artifact rules
- `framework/PROMPTS.md` — Prompt architecture

**Purpose:** Define the rules and principles that govern all other levels.

**Examples:**
- "Specs Before Code" (principle)
- "Templates are mandatory structure" (rule)
- "Agents MUST validate their output" (behavior constraint)

**Why this level exists:**
- Single source of truth for framework behavior
- All lower levels follow these rules consistently
- Changes to principles propagate systematically

**Output:** Rules that Level 1 must follow when defining structures.

## Level 1: Templates (Structure)

**What it contains:**
- `templates/specs/` — 5 specification templates (business, functional, stack, infrastructure, coverage)
- `templates/prompts/` — 2 prompt templates (specification-prompt, phase-prompt)
- `templates/agents/` — 2 agent templates (specification-agent, implementation-agent)

**Purpose:** Define the exact structure that Level 2 artifacts must follow.

**Examples:**
- Business spec template defines sections: Actors, Success Metrics, Use Case, Acceptance Criteria
- Agent template defines sections: Role, Input, Output, Directives, Completion Criteria
- Prompt template defines YAML frontmatter + suggested structure

**Why this level exists:**
- Reduces variance in Level 2 artifacts (consistency)
- Enables predictable consumption (downstream agents know structure)
- Separates "what structure" from "what content"

**Consumes:** Level 0 rules about template constraints, placeholder conventions, compliance requirements.

**Output:** Structures that Level 2 artifacts instantiate with actual content.

## Level 2: Agents & Artifacts (Instances)

**What it contains:**
- `agents/*.agent.md` — 8 agent definitions (business, functional, stack, infrastructure, coverage, development, deployment, validation)
- `prompts/*.prompt.md` — 8 prompt files (5 layer prompts + 3 phase prompts)
- `specs/**/*.md` — Specification documents (generated by agents in user projects)

**Purpose:** Concrete implementations that follow Level 1 templates and Level 0 rules.

**Examples:**
- `agents/smaqit.business.agent.md` follows `templates/agents/specification-agent.template.md`
- `prompts/smaqit.business.prompt.md` follows `templates/prompts/specification-prompt.template.md`
- User-generated business spec follows `templates/specs/business.template.md`

**Why this level exists:**
- Templates are abstract; agents/specs are concrete
- Each instance serves a specific purpose (business vs functional vs stack)
- Users interact with this level (invoke agents, fill prompts, review specs)

**Consumes:**
- Level 1 templates (structure to follow)
- Level 0 rules (behaviors to implement)

**Output:** Specifications and validated artifacts that Level 3 implements.

## Level 3: Application (Output)

**What it contains:**
- The built system (code, configurations, running infrastructure)

**Purpose:** Working software that satisfies all specifications from Level 2.

**Examples:**
- Source code implementing functional specs
- Deployed infrastructure matching infrastructure specs
- Test results validating coverage specs

**Why this level exists:**
- Ultimate output of the framework
- Measurable value (working software)
- Validates that all upstream levels are correct

**Consumes:**
- Level 2 specifications (requirements to satisfy)
- Level 0 principles (implementation guidance)

**Output:** Validated behavior meeting acceptance criteria.

## Level Dependencies

### Dependency Flow

```
Level 0 (Framework)
   ↓ [defines rules for]
Level 1 (Templates)
   ↓ [provides structure for]
Level 2 (Agents & Artifacts)
   ↓ [specifies requirements for]
Level 3 (Application)
```

### What "Consumes" Means

When Level N consumes Level N-1:
- **Agents** at Level 2 read templates from Level 1 to know what structure to produce
- **Templates** at Level 1 follow rules from Level 0 about placeholders, sections, compliance
- **Application** at Level 3 implements specifications from Level 2

### Amendment Propagation

When a level changes, downstream levels must adapt:

| Change Location | Impact |
|-----------------|--------|
| Level 0 (Framework rule changed) | Level 1 templates updated → Level 2 agents regenerated |
| Level 1 (Template structure changed) | Level 2 agents/specs regenerated to match new structure |
| Level 2 (Spec content changed) | Level 3 application reimplemented to match new requirements |

Changes flow downward, never upward. Level 3 doesn't change Level 2 specs; users change specs explicitly.

## Why Four Levels

### Alternative: Flat Structure

**What if everything was at the same level?**
- Agents, templates, and specs all mixed together
- No clear dependency order
- Changes to any file could affect any other file
- Circular dependencies possible

**Cost:** Unpredictable cascading changes, difficult to reason about impact.

### Alternative: Three Levels

**What if we merged Level 1 and Level 2?**
- Templates and agents would be at the same level
- Unclear whether agents follow templates or vice versa
- Template changes wouldn't have explicit propagation path

**Cost:** Ambiguous dependency order, harder to maintain consistency.

### Why Four Works

**Level 0 (Principles):** Timeless rules that change rarely
**Level 1 (Structure):** Stable structure that evolves with framework
**Level 2 (Content):** Variable content that changes frequently
**Level 3 (Output):** Ephemeral output regenerated as needed

Each level has different change frequencies and different audiences.

## Practical Implications

### For Framework Developers

When updating smaqit:
1. Start at Level 0 if principles change
2. Update Level 1 templates to reflect new principles
3. Regenerate Level 2 agents from updated templates
4. Test with Level 3 application builds

Follow the hierarchy—don't skip levels.

### For Framework Users

When using smaqit in a project:
- **Read Level 0** to understand principles
- **Ignore Level 1** (agents consume templates automatically)
- **Interact with Level 2** (invoke agents, fill prompts, review specs)
- **Validate Level 3** (test the built application)

Users don't need to think about levels—they're implementation details.

### For Agent Developers

When creating new agents:
1. Choose appropriate Level 1 template (specification or implementation agent)
2. Follow Level 0 rules from AGENTS.md
3. Produce Level 2 artifact that follows template exactly
4. Validate against Level 0 completion criteria

The hierarchy guides agent development systematically.

## Trade-offs

**Benefits:**
- Clear dependency order (no circular references)
- Predictable impact analysis (changes flow downward)
- Systematic evolution (update levels in order)
- Separation of concerns (each level has distinct purpose)

**Costs:**
- More complex mental model (four levels to understand)
- Rigid structure (can't easily skip levels)
- Propagation overhead (Level 0 change affects everything)

The cost is intentional—the hierarchy prevents ad-hoc changes that create inconsistency.
